# Virgil - Verification Framework for Disciplined AI-Assisted Development

## Project Name Origin

**Virgil** - Named after the Roman poet who guides Dante through Hell and Purgatory in the *Divine Comedy*.

Why this name fits:
- **Guide** - Virgil guides Dante through dangerous territory, just as this framework guides developers through the chaos of AI-assisted coding
- **Protector** - Virgil shields Dante from dangers, just as this framework protects codebases from unverified changes
- **Teacher** - Virgil explains what Dante encounters, just as this framework explains its decisions and helps users learn
- **Companion** - Virgil is not a tool but a trusted collaborator, reflecting this project's collaborative philosophy

**For non-technical users:** "Virgil helps me build my app safely."

---

## Project Vision

Build an open-source framework that combines:
- **Natural language code generation** (using LLMs)
- **Systematic verification** (automated + configurable rules)
- **Security & compliance enforcement** (OWASP, GDPR, NIST, PCI-DSS, etc.)
- **Immutable audit trails** (compliance-ready decision logs)
- **Distributed execution** (scale from single machine to clusters)

**Purpose:** Enable secure, auditable AI-assisted development by enforcing disciplinary gates before code reaches production.

**Core Philosophy:** Speed + Safety are not opposites. They require *structured verification*.

---

## Problem Statement

### The Gap
Current AI-assisted development tools (ChatGPT, Claude Code, v0):
- Optimize for speed over verification
- Generate code without comprehensive impact analysis
- Lack configurable security/compliance rules
- Provide no audit trail for regulatory compliance
- Create "vibe coded" patterns that violate security principles

### The Solution
A framework that:
- **Enforces assessment-first** (propose → verify → approve before code)
- **Performs comprehensive verification** (all affected code, all dependencies, all implications)
- **Applies configurable rules** (user-defined or standard: OWASP, GDPR, NIST, PCI-DSS)
- **Creates immutable audit trails** (every decision logged, traceable)
- **Scales from laptop to cluster** (same binary, multiple deployment modes)

---

## Guiding Principles

1. **DRY (Don't Repeat Yourself)** → No duplicate verification logic
2. **KISS (Keep It Simple, Stupid)** → Simple beats clever; easy to understand and maintain
3. **Kerckhoffs's Principle** → Security through transparency; everything is open source
4. **Security by Default** → Unsafe is opt-in exception, not the norm
5. **Verification > Speed** → Ship verified code, not fast code

## Target Audience & Design Philosophy

### **Who This Is For**

**Two distinct user groups with different needs:**

#### **1. Inexperienced Users (Complete Beginners)**
- No development background or minimal experience
- Want to build applications using AI assistance
- Don't understand security implications
- Will likely not review generated code comprehensively
- Example: Someone wanting to build a dating app, forum, or admin panel

**Their risk:** Ship catastrophically insecure code that exposes millions of records (Firebase misconfiguration, hardcoded passwords, plaintext API keys, missing authentication)

**Framework protects them by:** Strong defaults that make dangerous patterns impossible without explicit knowledge

#### **2. Experienced Users (Developers, Architects, Security Engineers)**
- Deep knowledge of security, compliance, architecture
- Want to work faster without sacrificing safety
- Will review and understand verification results
- Need flexibility and customization
- Example: You, security researchers, infrastructure teams

**Their need:** Automation that respects their discipline and doesn't get in the way

**Framework enables them by:** Full control, customization, and transparency

### **Design Principle: Safety By Default, Power By Choice**

**For beginners (default behavior):**
- ✓ Strong security rules enabled automatically (can't be disabled easily)
- ✓ Common dangerous patterns blocked (hardcoded secrets, plaintext auth, Firebase misconfiguration)
- ✓ Educational feedback (explains WHY something is unsafe)
- ✓ Verification gates that can't be bypassed without work

**For experts (advanced options):**
- ✓ Can configure rule strictness
- ✓ Can add custom rules for domain-specific needs
- ✓ Can override with explicit audit trail (logged as decision)
- ✓ Can inspect all verification logic

**Same framework. Different defaults.**

### **Real-World Context**

The security community has documented repeated breaches from "vibe coded" AI projects:
- 22+ million records exposed (Firebase misconfiguration by AI users)
- 500,000+ PII leaked (missing authentication + configuration errors)
- API keys exposed publicly on websites (generated by AI, deployed by non-technical founders)
- Plaintext passwords stored on landing pages (AI generated, no review)

**This framework prevents these specific disasters through mandatory verification before deployment.**

## Architecture Overview

### High-Level Flow

```
┌─────────────────────────────────────┐
│  Input (CLI / Web / API)            │
│  - Task description                 │
│  - Context / codebase               │
└──────────────┬──────────────────────┘
               │
┌──────────────▼──────────────────────┐
│  Assessment Phase                   │
│  - Parse intent                     │
│  - Propose approach                 │
│  - Wait for user approval           │
└──────────────┬──────────────────────┘
               │
┌──────────────▼──────────────────────┐
│  Code Generation Phase              │
│  - Generate code (LLM-based)        │
│  - Apply code quality checks        │
└──────────────┬──────────────────────┘
               │
┌──────────────▼──────────────────────┐
│  Verification Pipeline (Parallel)   │
│  - Impact Analysis                  │
│  - Security Analysis                │
│  - Compliance Checking              │
│  - Code Quality Review              │
│  - Assumption Logging               │
└──────────────┬──────────────────────┘
               │
┌──────────────▼──────────────────────┐
│  Decision Gate                      │
│  - All verifications passed?        │
│  - Present results to user          │
│  - Allow approval/rejection         │
└──────────────┬──────────────────────┘
               │
┌──────────────▼──────────────────────┐
│  Audit & Output                     │
│  - Log decision + reasoning         │
│  - Provide verified code            │
│  - Generate compliance report       │
└─────────────────────────────────────┘
```

### Core Components

#### **Orchestrator (Go, gRPC)**
- Central coordinator
- Manages workflow DAG execution
- Handles state persistence
- Supports distributed workers

#### **Verification Pipeline**
- Pluggable verification blocks
- Run in parallel where possible
- Each block = independent concern (security, compliance, impact, etc.)

#### **Rules Engine**
- User-configurable rules (OWASP, GDPR, NIST, PCI-DSS, custom)
- Rules can be enabled/disabled per project
- Rules versioned with deprecation support

#### **Model Integration**
- ONNX Runtime (local inference)
- Optional fallback to API-based models (for complex reasoning)
- Graceful degradation on low-resource machines

#### **Audit & Persistence**
- Append-only decision log
- SQLite (single machine) or distributed cache (clustering)
- Export capabilities (compliance reports, decision trails)

#### **Interfaces**
- **CLI** (primary, for integrated workflows)
- **Web UI** (secondary, for visualization and team collaboration)
- **gRPC API** (for programmatic access)

### **Phase 3: Model Integration (2-3 weeks)**
**Objective:** Add intelligent reasoning

**Deliverables:**
1. **ONNX Runtime integration**
   - Load quantized model (Mistral 7B or similar)
   - Basic inference pipeline
   - Graceful degradation on low-resource machines

2. **Assessment phase intelligence**
   - Model-based approach proposal
   - Confidence scoring
   - Fallback when uncertain

3. **Optional API fallback (Either/Or Model)**
   - Claude/GPT integration (for complex reasoning) - **only if user doesn't use `learn`**
   - User chooses: Either local pattern learning (`--augment learning`) OR external API fallback (`--augment api`), not both
   - If `--augment learning` active: only local ONNX model used (no external API calls)
   - If `--augment api` active (default): can use Claude/GPT fallback when local model uncertain
   - Cost tracking (for API usage)
   - **Rationale:** Keeps private codebases private; no pattern leakage to external APIs

**Output:** Framework can use AI for reasoning, not just pattern matching, with user privacy control

**Decision Gate:** Does intelligent routing improve results? Is the cost acceptable?

### **Phase 4: Self Learning Loop (2-3 weeks)**
**Objective:** Framework learns from decisions and improves iteratively through two complementary mechanisms

---

#### **Part A: User-Driven Pattern Extraction**

**Objective:** Allow experienced users to teach framework their established patterns

**Deliverables:**
1. **Pattern extraction from user codebases**
   - `virgil learn <path-to-codebase>` command
   - Analyzes user's code to extract patterns (fast initial learning)
   - Identifies: authentication patterns, data handling patterns, validation patterns, code structure
   - User codebase is never stored; only patterns are extracted

2. **Pattern storage & retrieval (Encrypted by default)**
   - All pattern data encrypted with ChaCha20-Poly1305 (authenticated encryption, mandatory not optional)
   - Security is fundamental, not optional: User-controlled storage (local only, no external transmission without explicit user action)
   - Post-quantum migration path planned (Phase 8+: inspired by SimpleX PQDR approach, integrate Kyber or NIST-approved post-quantum standards when stable)
   - Pattern library indexed by domain (auth, validation, data handling, etc.)
   - **Team collaboration model:** Teams share repositories via Git; each developer runs `virgil learn` independently on shared repo to extract local patterns (no pattern transmission between users, simple and privacy-respecting)
   - Integration with verification pipeline

3. **Pattern-aware verification**
   - New code checked against learned patterns
   - Flagged when code deviates from user's established patterns
   - Provides consistency feedback

4. **Usage in code generation**
   - `virgil create <task> --augment learning` (uses learned patterns)
   - Generation informed by user's patterns
   - Results respect user's established code style + standards

**Output:** Framework immediately applies user's good practices to new code

---

#### **Part B: Self Learning Loop (Continuous Improvement)**

**Objective:** Framework learns from its own assessment decisions and improves over time

**How it works:**
1. **ANALYZE:** `virgil assess <change>` evaluates code against rules
2. **JOURNAL:** Decisions stored in SQLite Cipher + MEMORY.md
3. **EXTRACT LEARNING:** Patterns extracted from decision history
4. **APPLY:** Learned patterns inform future assessments (improves accuracy over time)

**Deliverables:**
1. **Decision journaling**
   - Store all assessment decisions with context (what was checked, why, outcome)
   - Encrypted storage in SQLite Cipher
   - Indexed for efficient querying

2. **Pattern extraction from decision history**
   - Periodically analyze assessment history
   - Identify successful patterns (rules that caught real issues)
   - Identify ineffective patterns (rules that were too strict/lenient)
   - Refine verification rules based on evidence

3. **Adaptive verification**
   - Framework adjusts rule confidence based on historical accuracy
   - Rules that consistently catch problems get higher priority
   - Rules that frequently false-alarm get tuned or deprioritized
   - Results improve over time as framework learns project context

4. **MEMORY.md feedback**
   - Summary of learned patterns written to MEMORY.md
   - Human-readable (for audit and transparency)
   - Updated each session

**Output:** Framework becomes smarter over time; verification quality improves with each use

---

**Combined Output (Part A + Part B):** 
Framework learns from both user's established code patterns (Day 1 jumpstart) and its own assessment history (continuous improvement over time)

**Decision Gate:** Do both mechanisms improve code quality effectively? Are users comfortable with both automated and user-directed learning?

```
1. Initialize framework: virgil init
2. Define your rules: virgil config --rules owasp,gdpr,custom
3. Create feature: virgil create "Add email verification"
   → Framework proposes approach
```
